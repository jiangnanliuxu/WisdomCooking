#### 3.3 接口详细

**POST /codes**

请求：

```json
{
  "phone": "13800000001",
  "scene": "register|login|reset_password"
}
```

响应：

```json
{
  "code": 200,
  "data": {
    "expires_in": 300,
    "scene": "register",
    "dev_code": "926556"   // 仅 dev-mode=true 时返回
  }
}
```

安全控制：同一手机号+场景 60 秒内不可重复发送，验证码有效期 5 分钟，最多 5 次错误尝试。

**POST /register**

请求：

```json
{
  "phone": "13800000001",
  "code": "926556",
  "password": "Test123456",
  "confirmPassword": "Test123456",
  "nickname": "美食爱好者"
}
```

响应：返回 `AuthTokenVo`（含 JWT Token、过期时间和 UserProfileVo）。

业务规则：
- 密码 BCrypt 哈希后存入 `cook_users.password_hash`
- 手机号唯一，重复注册拒绝
- 密码与确认密码必须一致
- 昵称必填，最长 30 字符
- 注册成功后该手机号+场景下所有未用验证码批量作废

**POST /login**

请求（密码登录）：

```json
{
  "phone": "13800000001",
  "password": "Test123456"
}
```

请求（验证码登录）：

```json
{
  "phone": "13800000001",
  "code": "436176"
}
```

业务规则：
- 优先验证码登录，未传验证码则走密码比对
- 密码使用 BCrypt 比对（`SecurityUtils.matchesPassword`）
- `normal` 和 `muted` 状态用户可登录，`banned` 和 `deleted` 不可登录
- 登录成功签发 JWT Token，720 分钟有效，存入 Redis

**POST /logout**

无请求体，需在 Header 携带 `Authorization: Bearer <token>`。

**POST /password/reset**

请求：

```json
{
  "phone": "13800000001",
  "code": "589409",
  "password": "NewPass123",
  "confirmPassword": "NewPass123"
}
```

业务规则：验证码校验 → 用户存在性检查 → BCrypt 新密码哈希 → 更新 `cook_users.password_hash`。

#### 4.3 接口详细

**GET /me**

响应：

```json
{
  "code": 200,
  "data": {
    "id": 3,
    "phone": "138****0010",
    "nickname": "更新昵称",
    "avatarUrl": "https://example.com/avatar.jpg",
    "gender": "男",
    "birthday": "1995-06-15",
    "region": "北京市朝阳区",
    "bio": "热爱美食的程序员",
    "status": "normal",
    "interestTags": ["川菜", "烘焙", "减脂餐", "日料"],
    "statsJson": "{}"
  }
}
```

安全处理：手机号中间 4 位脱敏（138****0010），password_hash 不返回。

**PUT /me**

请求（所有字段可选）：

```json
{
  "nickname": "更新昵称",
  "avatarUrl": "https://example.com/avatar.jpg",
  "gender": "男",
  "birthday": "1995-06-15",
  "region": "北京市朝阳区",
  "bio": "热爱美食的程序员"
}
```

业务规则：动态更新（仅更新非空字段），手机号不可编辑。

**PUT /me/interests**

请求：

```json
{
  "interestTags": ["川菜", "烘焙", "减脂餐", "日料"]
}
```

业务规则：最多 5 个标签，每个最长 20 字符，以 JSON 数组存入 `cook_users.interest_tags_json`，覆盖写入。

## 第二阶段：菜谱主链路（仅后端）

> 完成时间：2026-05-11

### 1. 开发范围

第二阶段只开发后端 `ruoyi-cook` 模块，不改用户端和管理端前端。核心目标是打通菜谱从草稿、提交审核、后台审核、公开展示到评论互动的主链路。

### 2. 使用的数据表

本阶段没有新增数据库表，直接使用首版压缩表设计中的以下业务表：

| 表名 | 用途 |
|------|------|
| `cook_recipes` | 菜谱主表，保存作者、当前线上版本、审核状态、上架状态和统计计数 |
| `cook_recipe_versions` | 菜谱版本表，保存草稿、待审核版本、历史版本、食材 JSON、步骤 JSON、小贴士 JSON、视频 JSON |
| `cook_comments` | 评论表，本阶段用于菜谱评论和回复 |
| `cook_content_interactions` | 统一互动表，本阶段承载菜谱点赞、收藏、分享和评论点赞 |
| `cook_admin_operation_logs` | 后台操作日志，本阶段记录菜谱审核、上下架、删除和触发转码 |
| `cook_users` | 关联作者昵称、头像和账号状态 |

### 7. 状态流转规则

| 场景 | 规则 |
|------|------|
| 新建菜谱 | 写入 `recipes(review_status=draft,publish_status=offline)`，并创建 `recipe_versions(version_no=1,status=draft)` |
| 提交审核 | 最新版本状态改为 `pending_review`，主表 `review_status=pending_review` |
| 审核通过 | 待审核版本改为 `published`，`cook_recipes.current_version_id` 指向该版本，`publish_status=online` |
| 审核驳回 | 待审核版本改为 `rejected`，保存 `reject_reason`，旧的 `current_version_id` 不变 |
| 已发布后编辑 | 如果最新版本就是当前发布版本，则创建 `version_no + 1` 的新草稿；审核通过前旧版本继续公开展示 |
| 撤回审核 | 待审核版本改为 `withdrawn` |
| 公开查询 | 只查询 `publish_status=online` 且 `current_version_id` 指向 `published` 版本的菜谱 |

### 8. 互动与评论规则

- 收藏/取消收藏更新 `cook_recipes.favorite_count`。
- 评论/删除评论更新 `cook_recipes.comment_count`。
- 禁言或非正常状态用户不能发表评论。
- 第二阶段评论只开放 `target_type=recipe`，动态评论留到第三阶段。

### 4. 业务实现说明

- 搜索历史和热门词使用 Redis 实现，避免首版额外拆 `search_histories` 表。
- 打卡生成动态时，新增社区动态统一进入 `pending_review`，保持“所有动态先审后发”的规则不被破坏。
- 用户禁言 / 封禁 / 解封会同步写入 `cook_user_penalties` 与 `cook_admin_operation_logs`。
- 轮播图、反馈、举报、群组治理等后台关键动作都会写入 `cook_admin_operation_logs`。
- 上传文件首版采用本机目录落盘，并写入 `cook_media_assets`；返回的 `url` 指向 `/api/v1/uploads/{mediaId}/raw` 文件流接口。

### 5. 本轮核实后仍存在的文档边界

以下 5 个文档接口没有在本轮“强行做成伪实现”，因为它们与当前基础开发约束存在设计冲突：

- `POST /api/admin/v1/recipes`
- `PUT /api/admin/v1/recipes/{id}`
- `POST /api/admin/v1/categories`
- `PUT /api/admin/v1/categories/{id}`
- `DELETE /api/admin/v1/categories/{id}`

原因：

- 菜谱当前是“用户创作 -> 提交审核 -> 管理端审核 / 上下架”的主线，管理端直接代替用户创建 / 编辑菜谱缺少明确的作者归属规则。
- 分类当前是固定分类配置，开发文档第 4 章已明确“首版不单独建分类表”，因此后台动态 CRUD 与现有压缩表设计冲突。
