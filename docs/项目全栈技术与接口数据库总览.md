### 2.1 AI 与 RAG 基础能力

- 对话模型默认走 `qwen-turbo`，当前默认 chat 模型配置在 `cook_ai_models.config_json` 中限制 `max_tokens=130`，用于缩短回答长度并提升响应速度。
- 普通问答与 RAG 问答都要求纯文本输出，后端会清理常见 Markdown 标记；前端最终展示和落库内容以清理后的文本为准。
- RAG 知识库读取 `ruoyi-modules/ruoyi-cook/storage/embeddingDocs`，文档元数据与分片关系写入 MySQL，向量检索使用 Elasticsearch。
- 用户端 AI 问答支持 SSE 流式输出；管理端提供 AI 模型、对话日志、识图日志和知识库文档管理。

### 2.2 用户端共享基础设施

- `user-app/src/utils/request.ts`
  - `composeApiUrl()`：统一拼接用户端 API Base URL
  - `request()`：统一封装 `uni.request`，自动带 `Authorization`
  - `uploadFile()`：统一封装普通图片、非 H5 视频兜底、语音上传
- `user-app/src/utils/media.ts`
  - `resolveAssetUrl()`：把相对资源地址转成完整 URL
  - `mediaIdToRawUrl()`：把媒体 ID 转为 `/api/v1/uploads/{id}/raw`
- `user-app/src/components/OssVideoUploader.vue`
  - H5 菜谱视频优先走 OSS multipart 直传，checkpoint 写入本地 storage 用于断点续传
- `user-app/src/components/HlsVideoPlayer.vue`
  - 优先播放 `hlsUrl`，不可用时回退 MP4 `url`
- `user-app/src/stores/auth.ts`
  - 统一缓存 `token` 和当前用户资料
  - 通过 `GET /api/v1/users/me` 刷新用户资料
- `user-app/src/services/websocket.ts`
  - H5 下通过 STOMP 订阅 `/topic/conversations/{conversationId}`，接收实时消息
- `user-app/src/utils/amap.ts`
  - 动态发布页用于地图选点、POI 选择和逆地理编码
  - 编辑资料页复用高德定位和逆地理编码，只保存省、市、区/县级地区文本

### 2.3 管理端共享基础设施

- `admin-web/src/utils/request.ts`
  - Axios 请求拦截器，按 URL 自动切换网关 Base URL 和 `ruoyi-cook` Base URL
  - 自动附带管理员 Token，`401` 时跳回登录页
- `admin-web/src/stores/auth.ts`
  - `POST /auth/login` 登录
  - `GET /system/user/getInfo` 拉取管理员资料、角色和权限
  - 路由守卫中统一校验登录态
- `admin-web/src/components/HlsVideoPlayer.vue`
  - 菜谱详情弹窗复用 HLS 播放能力，Safari/iOS 走原生 HLS，其他现代浏览器由 `hls.js` 接入 `video.js`

## 6. 关键业务流的接口协作关系

### 6.1 发布菜谱

1. `POST /api/v1/uploads/images`
   - 上传封面图和步骤图，先拿到 `mediaId`
2. H5 视频走 OSS 分片直传链路；非 H5 仍可走 `POST /api/v1/uploads/videos`
   - `POST /api/v1/uploads/videos/multipart/init` 申请上传会话和 STS 临时凭证
   - 浏览器使用 `ali-oss.multipartUpload()` 直传原始视频到 OSS，并把 checkpoint 写到本地 storage
   - `POST /api/v1/uploads/videos/multipart/complete` 通知后端校验 OSS 对象，写入视频媒体信息并异步转码
3. `POST /api/v1/recipes`
   - 用标题、分类、图文步骤、视频等创建草稿
4. `POST /api/v1/recipes/{id}/submit`
   - 把草稿正式送进后台审核队列

### 6.1.1 OSS 视频上传与 HLS 播放链路

1. 前端提交 `fileName`、`fileSize`、`contentType`、`fingerprint` 到 `multipart/init`。
2. 后端读取 `config/cook-oss.yml` 的 OSS/STS 配置，创建 `cook_media_upload_sessions` 业务会话，并返回 STS 临时凭证、OSS `objectKey`、分片大小和 `checkpointKey`。
3. H5 浏览器使用 `ali-oss` 直接向 OSS 上传 `uploads/videos/original/**` 原片，业务后端不承接大视频流量。
4. 前端上传完成后调用 `multipart/complete`，后端 HEAD 校验对象存在且大小一致，写入 `cook_media_assets(status=uploaded,url=原片地址)`。
5. 后端异步调用 FFmpeg：下载原片到临时目录，生成单清晰度 VOD HLS 的 `index.m3u8` 和 `.ts` 分片，再上传到 OSS `uploads/videos/hls/{mediaId}/`。
6. 转码状态按 `uploaded -> transcoding -> ready/failed` 流转；成功时写 `hls_url`，失败时把错误放进 `metadata_json.errorMessage`。
7. 如果 Bucket 允许公开读，`hls_url` 可以直接是 OSS/CDN 地址；当前本地验证 Bucket 禁止公共读，因此后端会把 `hlsUrl` 对外改写为 `/api/v1/uploads/{mediaId}/hls/index.m3u8`，由业务后端代理 playlist 和 `.ts` 分片。
8. 用户端菜谱详情和管理端菜谱详情都优先播放 `hlsUrl`；Safari/iOS 使用原生 HLS，其他支持 MSE 的浏览器使用 `hls.js`，播放控件由 `video.js` 统一。

### 6.2 编辑已发布菜谱

1. `GET /api/v1/recipes/{id}`
   - 读取当前菜谱详情，供编辑页回显
2. `PUT /api/v1/recipes/{id}`
   - 保存修改内容
3. `POST /api/v1/recipes/{id}/submit`
   - 重新提交审核

说明：这里不是直接覆盖线上版本，而是生成新的 `cook_recipe_versions` 版本，旧版本继续对外可见，直到新版本审核通过。

2. `POST /api/v1/posts`

### 6.8 AI 智能问答与 RAG 回答

1. `GET /api/v1/ai/recommended-questions`
   - AI 页初始化时读取全站用户高频提问前三名，不足三条时由后端补默认饮食问题。
2. `POST /api/v1/ai/chat/stream`
   - 用户发送问题后优先使用流式接口；后端先写入用户消息，再检索知识库并调用默认 `qwen-turbo`。
3. RAG 命中时

## 8. 当前实现边界与阅读建议

- 这份文档区分了“后端已提供”和“当前页面已接入”。阅读时不要把两者混为一谈。
- 涉及 `ruoyi-cook` 的联调重启走 `bash scripts/start-all.sh restart --skip-build`，需要重新打包时先执行对应 Maven 构建；启动脚本默认把 RAG Elasticsearch 指向本机 Docker 映射 `ELASTICSEARCH_URIS=http://127.0.0.1:19200`，避免与 `ruoyi-auth:9200` 冲突。状态确认走 `bash scripts/start-all.sh status`，用户端 H5 地址为 `http://127.0.0.1:5174/`。
- OSS/STS/FFmpeg 运行配置放在 `projetc/RuoYi-Cloud/config/cook-oss.yml`，该文件只用于本机或部署环境读取，已被 `.gitignore` 忽略；仓库内的 `config/cook-oss.example.yml` 只保存字段示例，不保存真实 AccessKey。
- 视频上传当前默认要求 RAM 用户能够 `AssumeRole` 到专用 OSS 上传角色；如果 `multipart/init` 返回 `NoPermission` 或 `not authorized`，优先检查 `sts-role-arn` 是否为可被当前 RAM 用户扮演的角色，以及该角色的信任策略和 OSS 权限策略。
- H5 浏览器直传 OSS 还要求 Bucket CORS 允许本地/部署域名、`GET/POST/PUT/DELETE/HEAD` 和常用请求头；未配置时浏览器会在 `?uploads=` 预检阶段报 `CORS is not enabled for this bucket`。
- 当前测试 Bucket 禁止公共读和 public object ACL；播放链路已采用后端 HLS 代理，不需要关闭 OSS 的公共访问保护。
- FFmpeg 路径由 `cook.media.ffmpeg-path` 控制，本机未安装或路径错误时转码任务会进入 `failed`，错误信息写入 `cook_media_assets.metadata_json` 并在管理端菜谱详情展示。
- 用户端当前没有“我的举报”独立页面，所以举报相关以“提交举报”为主，`GET /api/v1/reports*` 还没有真实页面落地。
- 管理端已有“用户反馈”独立 View，用于查看反馈列表、详情和截图；“举报管理”独立 View 尚未接入，但后端接口已经存在。
- 首页当前通过 `GET /api/v1/home` 一次聚合取数，不是分别调用轮播、推荐达人、热门词接口。
- 首页轮播当前由用户端消费 `/api/v1/home.banners`，支持自动切换和 H5 鼠标/触控左右拖拽；当后端有效轮播少于 2 张时，前端补明确占位卡用于保持轮播交互，不伪装为真实运营活动。
- 菜谱分类入口已经数据库化：管理端维护 `cook_categories`，新增分类会进入用户端分类页和发布菜谱分类选择；删除/禁用后用户端不再展示，已有菜谱记录不被删除。
- 首页和社区推荐流会读取当前用户兴趣标签；当前兴趣标签限定为八大中华菜系，旧的非菜系标签不会参与推荐排序。
- 社区广场顶部预选项仍固定为“推荐、关注、八大中华菜系、其他”，不跟随 `cook_categories` 的普通新增分类变化；八大菜系在管理端只读，后端也拒绝修改和删除。
- 社区广场“关注”筛选当前不伪造关注流数据，后端没有独立关注动态流接口前仅作为后续接入口保留。
- 社区动态卡片的关联菜谱名称来自 `GET /api/v1/posts*` 返回的 `relatedRecipeTitle`，该字段由后端 `PostMapper` 联表 `cook_recipes.current_version_id -> cook_recipe_versions.title` 得到；如果运行包仍只返回 `relatedRecipeId`，优先重新打包并重启 `ruoyi-cook`，不要在前端逐条请求菜谱详情兜底。
- 消息系统当前是“REST 拉历史 + STOMP 推实时”的组合：拉列表和发消息靠 REST，实时新消息靠 WebSocket 订阅。
- AI 对话当前默认 chat 模型为 `qwen-turbo`，当前运行库把 `max_tokens` 限制为 130；后续重建环境时如果需要自动生成该默认模型，需要额外补模型种子或迁移脚本。
- AI RAG 的检索上下文仍使用知识库分片正文，但用户端和管理端来源区域只展示 `title` 或 `fileName`；`sources[*].snippet` 已不作为前端展示内容。
