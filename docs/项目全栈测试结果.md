## 1. 测试范围

本轮按全栈技术文档中的接口和流程，对用户端、管理端、`ruoyi-cook` 后端和本地运行接口进行验证。

覆盖内容：

- 用户端页面接口：认证、首页、搜索、菜谱、社区、评论、上传、打卡、反馈、消息、AI 历史接口。
- 管理端页面接口：登录依赖的若依基础接口、`/api/admin/v1` 运营、审核、AI、用户、群组、轮播图接口封装。
- 后端控制器：`ruoyi-cook` 下所有 `*Controller.java` 映射。
- 关键流程：登录、菜谱发布审核、动态先审后发、打卡、反馈、媒体上传、消息/AI 列表读取。

补测后仍未完整自动化覆盖：

- AI 问答和 AI 识图真实调用：会触发外部模型能力，且依赖真实 API Key/可能计费，本轮只验证历史列表接口。
- WebSocket STOMP 实时推送：本轮验证 REST 历史消息和通知列表，未建立 STOMP 客户端订阅。

### 7.3 网关路由核对

同一管理员 Token 访问 `http://127.0.0.1:8080/api/admin/v1/**` 时，业务接口返回：

```text
404 NOT_FOUND "No static resource api/admin/v1/..."
```

这说明当前网关配置只覆盖了若依基础路由，例如 `/auth/**`、`/system/**`、`/file/**`，没有把 `/api/**` 或 `/api/admin/v1/**` 路由到 `ruoyi-cook`。管理端开发环境通过 Vite 代理直连 `9210` 可以绕开该问题，但生产如果只部署在网关同源下，`/api/admin/v1/**` 会失败。

## 8. 发现的问题与风险

| 等级 | 问题 | 影响 |
| --- | --- | --- |
| P1 | 未登录访问 `GET /api/v1/users/me` 返回业务码 `500`，而不是统一的 `401` | 已修复：`NotLoginException` 统一返回业务码 `401`，复测未登录访问返回 `{"msg":"未提供token","code":401}` |
| P1 | 网关缺少 `/api/**` 或 `/api/admin/v1/**` 到 `ruoyi-cook` 的路由 | 已处理：网关 Nacos 配置已补充 `/api/** -> ruoyi-cook` 路由，`/api/v1/**` 允许进入业务服务，由业务层统一处理登录态 |
| P2 | `/api/v1/auth/codes` 当前环境不返回 `dev_code` | 无短信通道时，注册、验证码登录、找回密码闭环无法自动化验证；密码登录可用 |
| P2 | `admin-web/.env.production` 当前只配置了 `VITE_ADMIN_API_BASE=/prod-api`，未配置 `VITE_COOK_API_BASE` | 生产构建下 `/api/admin/v1/**` 默认同源访问；如果部署入口没有反代或网关路由，会复现 404 |
| P3 | `user-app` H5 构建有 Sass legacy JS API deprecation warning | 当前不阻塞构建，但后续 Dart Sass 2.0 可能需要升级依赖或构建链 |
| P3 | `admin-web` 构建提示主 chunk 超过 500KB | 当前不阻塞构建；后续可按路由拆分或启用更细的 code splitting |

## 10. 结论

本轮测试结论：项目主干全栈链路可运行。全栈文档中的 `ruoyi-cook` 业务接口均能在后端控制器中找到映射，用户端和管理端构建通过，后端编译通过，本地 `9210` 后端服务可用，用户端核心流程烟测通过。补测后确认管理端基础登录链路可通过 `8080` 网关完成，管理端业务接口也能通过 `8080` 网关进入鉴权/业务链路，不再返回网关 404。

上线或交付前状态：

1. P1：未登录/Token 缺失业务码已统一为 `401`，与前端拦截器保持一致。
2. P1：网关已补充 `/api/**` 到 `ruoyi-cook` 的路由，管理端业务接口不再通过网关返回 404。
3. P2：在可控测试 Key 环境下补跑 AI 问答、AI 识图真实模型调用。
4. P3：清理或标记本轮保留的媒体测试数据；反馈测试数据已通过管理端标记为 `closed`。
