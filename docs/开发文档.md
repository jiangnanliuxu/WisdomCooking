# 码上智厨开发文档

> 本文面向工程实现、任务拆分、接口联调和后续迭代维护。产品细节以 [需求文档.md](需求文档.md) 为准，初始化上下文以 [INIT.md](INIT.md) 为准。

## 1. 项目概览

码上智厨是一款面向美食学习、菜谱分享、社区互动、饮食打卡和 AI 营养服务的应用，包含用户端、管理端和后端服务。

核心能力：

- 用户端：注册登录、验证码认证、首页推荐、搜索分类、菜谱详情、HLS 教学视频、菜谱发布审核、社区动态、评论互动、收藏点赞、打卡月历、AI 营养问答、拍照识热量、私信、群聊、通知、反馈和举报。
- 管理端：仪表盘、菜谱管理、分类管理、轮播图管理、菜谱审核、动态审核、用户管理、群组管理、内容举报处理、AI 模型配置、AI 对话日志、AI 识图日志。
- 后端：基于 RuoYi-Cloud v3.6.8 骨架开发，业务代码集中在 `ruoyi-modules/ruoyi-cook` 模块，复用若依网关、认证、系统管理、文件、定时任务等基础能力。

首版基础开发要求：

1. 用户端必须包含独立登录、注册、验证码登录和找回密码能力。
2. 已发布菜谱编辑后生成新版本并重新审核，审核通过前旧版本继续展示。
3. 所有动态先审后发。
4. AI 问答与 AI 识图分别使用对话模型和视觉模型。
5. 用户可在消息中心右上角"＋"自主创建群聊。
6. 消息中心固定包含私信、群聊、通知三个 Tab。
7. 菜谱详情、动态详情、评论项都必须有独立举报入口。
8. 编辑资料页的性别、生日、地区支持直接编辑。
9. 菜谱分类固定分为中华菜系、场景分类、更多分类。
10. 打卡月历支持点击日期查看当天全部打卡。
11. 发布菜谱页底部按钮为"提交"，提交后进入管理端审核列表，审核通过后才公开发布。

## 2. 总体架构

### 2.1 端侧划分

| 端 | 说明 | 主要职责 |
| --- | --- | --- |
| 用户端 | uniapp 应用 | 面向普通用户，覆盖浏览、发布、互动、消息、AI、反馈和举报 |
| 管理端 | Vue 3 管理后台 | 面向运营、内容审核、社区管理、AI 运维和超级管理员 |
| 后端 | RuoYi-Cloud v3.6.8（`projetc/RuoYi-Cloud/`），业务模块 `ruoyi-cook` | 提供用户端 API、管理端 API、WebSocket、AI 调用、文件处理和审计 |
| 基础设施 | MySQL、Redis、OSS、FFmpeg、AI Provider | 数据持久化、缓存、实时状态、媒体存储、视频转码和模型能力 |

### 2.2 后端架构：基于 RuoYi-Cloud + ruoyi-cook

后端基于已就绪的 RuoYi-Cloud v3.6.8 骨架（位于 `projetc/RuoYi-Cloud/`）开发。若依 Cloud 已提供以下基础设施模块，直接复用：

| 若依内置模块 | 复用方式 |
| --- | --- |
| `ruoyi-gateway` | API 网关，统一入口，负责路由、限流、鉴权 |
| `ruoyi-auth` | 认证中心，Token 签发与校验 |
| `ruoyi-system` | 系统管理，用户/角色/菜单/字典/参数配置 |
| `ruoyi-gen` | 代码生成器，辅助生成 CRUD 模板 |
| `ruoyi-job` | 定时任务调度（xxl-job），用于数据校准、统计聚合、热门词刷新 |
| `ruoyi-file` | 文件服务，对接 OSS/MinIO，提供统一上传接口 |
| `ruoyi-monitor` | 监控中心，接口耗时、在线状态等可观测性 |

在 `ruoyi-modules` 下新建 `ruoyi-cook` 作为**独立微服务模块**。它拥有自己的启动类、application.yml、独立端口，注册到 Nacos，与 `ruoyi-system`、`ruoyi-job` 等模块平级——只是首版将所有业务域合并在这一个微服务内，内部按包分层，降低拆分成本。`ruoyi-modules` 仅作为 Maven 聚合父 POM 存在，不做业务耦合。`ruoyi-cook` 内部按业务域分包：

| 包路径 | 职责 |
| --- | --- |
| `**.cook.auth` | 用户端注册、登录、验证码、密码重置；复用若依认证，扩展业务鉴权 |
| `**.cook.user` | 用户资料、兴趣标签、关注关系、禁言封禁 |
| `**.cook.recipe` | 菜谱、分类、版本、食材、步骤、视频、菜谱审核 |
| `**.cook.community` | 动态、话题、动态审核、动态屏蔽 |
| `**.cook.interaction` | 评论、点赞、收藏、分享统一互动 |
| `**.cook.checkin` | 打卡记录、月历、打卡生成动态 |
| `**.cook.message` | 私信、群聊、通知、WebSocket 推送 |
| `**.cook.group` | 群组、成员、公告、群管理 |
| `**.cook.operation` | 轮播图、搜索、反馈、举报 |
| `**.cook.ai` | AI 模型配置、AI 问答、AI 识图、日志和内容标记 |
| `**.cook.media` | 文件上传接口封装，视频转码任务 |
| `**.cook.audit` | 管理员操作日志、审核日志、处罚记录、统计聚合 |

> 后续数据量或团队规模上来后，可将 `ruoyi-cook` 内部包拆分为多个独立微服务模块（如 `ruoyi-cook-recipe`、`ruoyi-cook-community` 等），当前包边界已为此预留。

## 4. 数据库设计

### 4.1 通用约定

数据库使用 MySQL 8.0。所有业务表默认包含：

| 字段 | 说明 |
| --- | --- |
| `id BIGINT PRIMARY KEY` | 主键，建议雪花 ID 或数据库自增策略统一 |
| `created_at DATETIME NOT NULL` | 创建时间 |
| `updated_at DATETIME NOT NULL` | 更新时间 |
| `deleted_at DATETIME NULL` | 逻辑删除时间，需要逻辑删除的表使用 |

开发约定：

- 金额、计数、状态字段不使用中文值，统一用枚举码。
- 用户输入内容需要做敏感词、XSS 和长度校验。
- 关键操作写入审计日志，包括审核、封禁、解禁、模型配置、Prompt 修改、轮播上下架。
- 点赞数、收藏数、评论数、粉丝数可冗余存储，但需要异步校准任务。
- 公开内容查询必须过滤删除、下架、未审核、被屏蔽状态。

### 4.2 状态枚举

| 枚举 | 取值 |
| --- | --- |
| `user_status` | `normal` 正常、`muted` 禁言、`banned` 封禁、`deleted` 注销 |
| `recipe_review_status` | `draft` 草稿、`pending_review` 审核中、`published` 已通过、`rejected` 已驳回、`withdrawn` 已撤回 |
| `recipe_publish_status` | `online` 已上架、`offline` 已下架 |
| `video_status` | `none` 无视频、`uploaded` 已上传、`transcoding` 转码中、`ready` 已就绪、`failed` 失败 |
| `post_status` | `draft` 草稿、`pending_review` 审核中、`published` 已通过、`rejected` 已驳回、`blocked` 已屏蔽、`withdrawn` 已撤回 |
| `feedback_status` | `processing` 处理中、`resolved` 已解决、`closed` 已关闭 |
| `group_status` | `normal` 正常、`warned` 已警告、`dissolved` 已解散 |
| `report_status` | `pending` 待处理、`accepted` 已采纳、`rejected` 已驳回、`closed` 已关闭 |
| `ai_model_type` | `chat` 对话模型、`vision` 视觉模型 |
| `ai_flag` | `normal` 正常、`warning` 警告、`violation` 违规 |
| `recognition_status` | `success` 识别成功、`failed` 识别失败、`unknown` 无法识别 |

### 4.3 首版压缩表设计

首版按"少表、清晰、可拆"原则设计，建议控制在 22 张业务表左右。若依后台自带的系统用户、角色、菜单、权限表不计入业务表数量；如果不使用若依内置权限表，再单独补一套后台账号表。

#### 压缩后的业务表

| 表 | 归属 | 说明 |
| --- | --- | --- |
| `cook_users` | 用户 | 普通用户资料、账号状态、兴趣标签 JSON、第三方绑定 JSON、统计计数 |
| `cook_verification_codes` | 用户 | 短信验证码，包含手机号、场景、验证码、过期时间、使用状态、错误次数 |
| `cook_recipes` | 菜谱 | 菜谱主表，保存当前线上版本、审核状态、上架状态和统计计数 |
| `cook_recipe_versions` | 菜谱 | 菜谱版本表，承载草稿、待审核版本和历史版本；食材、步骤、小贴士、视频信息用 JSON 或 `media_id` 关联 |
| `cook_posts` | 社区 | 动态主表，图片、话题、位置、关联菜谱等用 JSON 或 `media_id` 关联 |
| `cook_comments` | 互动 | 菜谱、动态评论，支持一级评论和二级回复 |
| `cook_content_interactions` | 互动 | 点赞、收藏、关注、分享等统一互动表 |
| `cook_checkins` | 打卡 | 用户饮食打卡记录，图片和来源信息用 JSON 或 `media_id` 关联 |
| `cook_groups` | 消息 | 群组基础信息、群公告、群状态 |
| `cook_conversations` | 消息 | 私信、群聊、通知三类会话，保存最后消息和会话状态 |
| `cook_conversation_members` | 消息 | 会话成员、群成员角色、未读数、置顶、免打扰、入群时间 |
| `cook_messages` | 消息 | 私信、群聊、通知消息统一存储，内容体使用 JSON 支持文本、图片、语音、系统通知 |
| `cook_banners` | 运营 | 首页轮播图、跳转目标、有效期、曝光和点击数据 |
| `cook_feedbacks` | 运营 | 产品问题反馈和后台回复 |
| `cook_reports` | 风控 | 内容举报，支持菜谱、动态、评论、用户对象 |
| `cook_ai_models` | AI | 对话模型和视觉模型配置、参数、默认启用状态 |
| `cook_ai_messages` | AI | AI 对话消息和会话归档，使用 `conversation_id` 分组，不单独建会话表 |
| `cook_ai_image_recognition_logs` | AI | AI 识图日志，识别结果、营养结构、候选结果用 JSON |
| `cook_media_assets` | 文件 | 图片、视频、音频上传资源统一表 |
| `cook_user_penalties` | 审计 | 用户警告、禁言、封禁处罚记录 |
| `cook_admin_operation_logs` | 审计 | 管理员操作、审核、模型配置、Prompt 修改、轮播上下架等日志 |
| `cook_daily_statistics` | 统计 | 每日统计指标；首版也可先由实时聚合替代，数据量上来后再启用 |

#### 不单独建表的内容

| 原设计对象 | 首版承载方式 | 拆表时机 |
| --- | --- | --- |
| `recipe_categories` | 固定分类配置或后端枚举，接口按中华菜系、场景分类、更多分类返回 | 运营需要后台动态维护分类层级、图标、排序时拆表 |
| `recipe_ingredients`、`recipe_steps` | 存入 `cook_recipe_versions.ingredients_json`、`cook_recipe_versions.steps_json` | 需要按食材检索、步骤级复用或精细统计时拆表 |
| `recipe_videos` | 视频 `media_id`、HLS 地址和转码状态存入 `cook_recipe_versions.video_json` 或 `cook_media_assets` | 视频转码任务复杂、需要多清晰度和多版本管理时拆表 |
| `recipe_audit_logs`、`post_audit_logs` | 写入 `cook_admin_operation_logs`，使用 `biz_type`、`biz_id`、`action` 区分 | 审核日志查询量大、需要独立报表时拆表 |
| `topics`、`post_topics` | 动态话题存入 `cook_posts.topic_codes_json` | 话题运营、话题页、热榜和订阅关系复杂时拆表 |
| `post_media` | 动态媒体存入 `cook_posts.media_ids_json`，资源详情在 `cook_media_assets` | 动态媒体需要单独审核、排序或统计时拆表 |
| `group_members` | 合并到 `cook_conversation_members`，群聊会话成员即群成员 | 群组织关系、群管理员体系复杂时拆回独立表 |
| `group_notices` | 当前群公告存入 `cook_groups.notice`，历史公告写 `cook_messages` 系统消息 | 群公告需要多条历史和单独管理时拆表 |
| `notifications` | 通知作为 `cook_conversations.type=notification` 下的 `cook_messages` | 通知需要独立模板、批量投递和复杂状态时拆表 |
| `search_histories`、`hot_search_keywords` | 首版用 Redis 或用户本地缓存，热门词由日志聚合 | 需要长期搜索分析和运营配置时拆表 |
| `ai_conversations` | `cook_ai_messages.conversation_id` 分组，首条消息冗余标题和用户信息 | AI 会话列表、归档、分享和删除策略复杂时拆表 |
| `ai_conversation_marks` | 标记字段直接放在 `cook_ai_messages` | 需要多管理员复核、多次标记历史时拆表 |

关键规则：

- 手机号唯一；注册、登录、找回密码使用同一套验证码逻辑。
- 禁言用户不能发动态、评论、私信和群消息；封禁用户不能登录。
- 已发布菜谱再次编辑时，新内容写入 `cook_recipe_versions`，审核通过后替换线上版本。
- 所有动态先审后发，审核和屏蔽动作统一写入 `cook_admin_operation_logs`。
- 点赞、收藏、关注和分享使用 `cook_content_interactions` 幂等记录。
- 反馈和举报分表处理，不能混用。
- AI 问答与 AI 识图模型配置分开，每次调用都写日志。
- 文件上传接口统一返回 `media_id`、`url`、文件元信息。
- 统计表不是首版强依赖，可先实时聚合，后续用定时任务沉淀到 `cook_daily_statistics`。

### 4.4 核心关系

- 用户 `1:N` 菜谱、动态、评论、打卡、消息、反馈、举报。
- 用户兴趣标签首版存 `cook_users.interest_tags_json`；关注关系通过 `cook_content_interactions(target_type=user, action_type=follow)` 承载。
- 分类首版使用固定配置；菜谱保存分类编码。
- 菜谱 `1:N` 菜谱版本；食材、步骤、小贴士和视频信息随版本存储。
- 食材、步骤、视频首版存入 `cook_recipe_versions` 的 JSON 字段或关联 `cook_media_assets`。
- 动态与话题首版通过 `cook_posts.topic_codes_json` 关联。
- 菜谱、动态、评论、用户与点赞收藏关注等互动 `N:N`，通过 `cook_content_interactions` 统一记录。
- 群组与用户 `N:N`，首版通过 `cook_conversation_members` 统一承载。
- 会话与用户 `N:N`，通过 `cook_conversation_members` 关联。
- 会话 `1:N` 消息。
- 通知作为特殊会话和系统消息处理。
- AI 模型 `1:N` AI 对话消息，模型类型为 `chat`。
- AI 模型 `1:N` AI 识图记录，模型类型为 `vision`。
- 举报 `N:1` 指向菜谱、动态、评论或用户。

## 5. 接口设计

### 5.1 通用约定

接口前缀：

| 端 | 前缀 |
| --- | --- |
| 用户端 | `/api/v1` |
| 管理端 | `/api/admin/v1` |

认证约定：

- 用户端使用 Bearer Token。
- 管理端使用管理员 Token。
- 登录、注册、发送验证码、公开内容浏览可按白名单放行。

响应约定：

```json
{
  "code": 0,
  "message": "success",

}
```

```json
{
  "page": 1,
  "page_size": 20,
  "total": 100,
  "items": []
}
```

错误约定：

- `400` 参数错误。
- `401` 未登录或 Token 失效。
- `403` 无权限或账号被禁用。
- `404` 资源不存在。
- `409` 幂等冲突或状态冲突。

- 服务端写库成功后再推送给接收方。

## 6. 业务流程

### 6.1 登录注册与验证码

1. 用户输入手机号和场景，请求发送验证码。
2. 后端校验手机号格式、发送频控、场景合法性。
3. 写入 `cook_verification_codes`，发送短信。
4. 注册、验证码登录或找回密码时校验手机号、验证码、过期时间、错误次数和使用状态。
5. 注册成功写入 `cook_users`；登录成功签发 Token。
6. 验证码使用后标记为已使用。

### 6.2 菜谱发布审核

1. 用户填写菜谱并保存草稿，状态为 `draft`。
2. 用户点击"提交"，校验封面、名称、分类、难度、耗时、简介、至少 1 个食材、至少 1 个步骤。
3. 菜谱状态变为 `pending_review`，进入管理端菜谱审核列表。
4. 管理员查看详情，包含封面、简介、食材、步骤、视频、小贴士和提交用户。
5. 审核通过后状态变为 `published`，公开展示；有视频则触发转码任务。
6. 审核驳回后状态变为 `rejected`，写入驳回原因和补充说明，并通知用户。
7. 用户在我的菜谱页查看驳回原因，修改后重新提交。

### 6.3 已发布菜谱编辑

1. 用户编辑已发布菜谱。
2. 后端不直接覆盖线上内容，而是生成 `cook_recipe_versions` 新版本。
3. 新版本进入 `pending_review`。
4. 审核通过后替换 `cook_recipes.current_version_id`。
5. 审核未通过时旧版本继续在线展示。

### 6.4 动态发布与审核

1. 用户发布动态，包含文本、图片、可选关联菜谱、话题、位置和可见范围。
2. 动态初始状态为 `pending_review`，审核通过前不进入社区公开流。
3. 管理员审核通过后状态为 `published`。
4. 管理员屏蔽动态时选择违规原因和处理方式。
5. 若处理方式包含警告或禁言，生成用户处罚记录。
6. 屏蔽或恢复操作写入动态审核日志并发送通知。

### 6.5 评论、点赞和收藏

1. 用户对菜谱、动态发表评论，评论可为一级评论或二级回复。
2. 作者回复展示作者标识。
3. 点赞和收藏通过 `cook_content_interactions` 统一记录。
4. 同一用户对同一对象同一互动类型只能有一条有效记录。
5. 重复点击执行取消操作。
6. 评论项提供独立举报入口。

### 6.6 打卡

1. 用户选择菜谱或自由记录饮食，上传打卡图片和心得。
2. 写入 `cook_checkins`。
3. 月历接口按月份返回已打卡日期。
4. 用户点击某日期时，请求 `/api/v1/checkins/by-date` 查看当天全部打卡。
5. 用户可将打卡一键生成动态，生成后的动态进入审核流程。
6. 删除打卡不自动删除已生成动态，只解除关联或保留来源标识。

### 6.7 消息和群聊

1. 用户进入消息页后建立 WebSocket 连接。
2. 客户端按 Tab 拉取私信、群聊或通知列表。
3. 用户点击消息中心右上角"＋"创建群聊，创建者成为群主。
4. 进入会话后分页拉取历史消息。
5. 用户发送消息，后端校验禁言状态、会话成员权限和群成员关系。
6. 消息入库后推送给接收方，并更新会话最后消息和未读数。
7. 接收方读消息后发送已读回执，服务端清空或减少未读数。

### 6.8 AI 问答

1. 用户输入饮食问题或点击快捷问题。
2. 后端读取当前默认启用的对话模型和系统 Prompt。
3. 调用模型，可通过 WebSocket 或 SSE 返回流式内容。
4. 记录用户输入、AI 输出、模型、Token、耗时和会话 ID。
5. 管理员可在对话日志中查看并标记正常、警告或违规。

### 6.9 AI 识图

1. 用户拍照或从相册选择食物图片。
2. 后端保存图片资源，调用当前默认启用的视觉模型。
3. 返回菜品名称、置信度、热量、蛋白质、脂肪、碳水、膳食纤维、分析建议和候选结果。
4. 写入 `cook_ai_image_recognition_logs`。
5. 用户端识图结果展示免责声明。
6. 管理端可按状态、时间、用户、菜品查询识图日志。

### 6.10 内容举报

1. 用户在菜谱详情、动态详情或评论项点击举报入口。
2. 用户选择举报类型，填写举报原因、补充说明，并可上传截图。
3. 系统生成 `cook_reports` 记录，初始状态为 `pending`。
4. 管理员查看举报详情，包含被举报内容快照、举报用户、对象关联信息。
5. 管理员可采纳、驳回或关闭。
6. 采纳后可联动屏蔽内容、禁言、封禁或转入对应审核处理。

### 6.11 文件上传与视频转码

1. 用户端或管理端上传图片、音频或视频。
2. 后端写入 `cook_media_assets`，返回 `media_id` 和访问 URL。
3. 大视频建议分片上传，后端记录 `uploadId` 和分片索引。
4. OSS 完成合并后触发转码任务。
5. 视频状态依次为 `uploaded`、`transcoding`、`ready` 或 `failed`。
6. 转码成功后生成 `.m3u8` 和 `.ts` 切片，播放器使用 HLS 地址。
7. 转码失败可重试，必要时降级使用原始 MP4。

## 7. 权限与安全

### 7.1 角色权限

| 角色 | 权限范围 |
| --- | --- |
| 游客 | 查看首页、分类、搜索、公开菜谱与社区内容 |
| 普通用户 | 发布菜谱和动态、点赞、收藏、评论、关注、打卡、反馈、私信、群聊、AI 问答与识图 |
| 内容管理员 | 管理菜谱、分类、轮播图，审核菜谱和动态，处理反馈和举报 |
| 社区管理员 | 管理用户状态、群组、动态违规处理、消息与社区秩序 |
| AI 运维管理员 | 配置 AI 模型参数，查看对话日志和识图日志，标记违规内容 |
| 超级管理员 | 拥有全部后台权限和管理员账号管理能力 |

### 7.2 安全要求

- 密码使用强哈希算法存储。
- API Key 加密存储，前端只展示掩码。
- 所有管理端操作必须鉴权并写审计日志。
- 用户输入需要做长度限制、敏感词过滤和 XSS 过滤。
- 文件上传校验类型、大小、扩展名和 MIME。
- 举报、验证码、评论、点赞等接口需要防刷和幂等控制。
- AI 高风险问题需要拒答或安全引导，不能替代专业医疗建议。

## 8. 非功能需求

| 合规 | AI 建议和识图结果需展示免责声明 |
| 可恢复 | 视频转码失败可重试，关键业务操作保留日志 |

## 10. 实现注意事项

- 不要把反馈和举报合并：反馈是产品问题，举报是内容风控。
- 不要让已发布菜谱直接覆盖线上内容：必须走 `cook_recipe_versions` 审核。
- 不要让动态先发后审：所有动态必须先审后发。
- 不要让 AI 识图复用对话模型默认配置：视觉模型必须独立。
- 菜谱分类分组固定，不需要后台维护分组字典。
- 通知保留在消息中心 Tab 内，不需要独立通知中心。
- 发布菜谱页底部按钮文案是"提交"，右上角无"发布"按钮。
- 删除建议优先逻辑删除，保留历史互动和审计。
- 对外公开接口必须统一过滤下架、删除、审核中、被屏蔽内容。
- 后台所有影响内容展示、用户状态和模型配置的操作都必须记录操作人和时间。
- Nacos 配置变更必须通过 Nacos 控制台（`http://127.0.0.1:8848/nacos`）或 Open API 进行，严禁直接修改 MySQL `config_info` 表。直接 SQL 修改不会被运行中的 Nacos 感知，需重启 Nacos 容器才能生效，运维成本高且容易引发配置不一致。
